# Задание 4 — Обоснование конфигурации Terraform и облачной инфраструктуры

## 1. Общая логика архитектуры

Выбран подход **разделения сети на публичную и приватную зоны** с использованием VPC:

- **Публичная подсеть** — для сервисов, которые должны быть доступны извне (портал самообслуживания).  
- **Приватная подсеть** — для критичных сервисов данных (DWH, оркестратор и т. п.), доступных только по внутренним каналам.

Такое разделение:

- Повышает безопасность: база данных и внутренние сервисы недоступны напрямую из интернета.  
- Упрощает соблюдение требований регуляторов для финансовых и медицинских данных.

## 2. Сетевой уровень: VPC, подсети, NAT

- **VPC (`yandex_vpc_network.main`)**  
  - Позволяет логически изолировать инфраструктуру проекта «Будущее 2.0».  
  - CIDR выбирается так, чтобы не конфликтовать с локальными сетями и возможными VPN.

- **Публичная подсеть (`yandex_vpc_subnet.public`)**  
  - Разворачивает портал самообслуживания, доступный пользователям через интернет.  
  - В Yandex Cloud публичные подсети автоматически получают доступ в интернет через NAT.

- **Приватная подсеть (`yandex_vpc_subnet.private`)**  
  - Размещает доменное хранилище (Managed PostgreSQL) и другие сервисы данных.  
  - Доступ наружу осуществляется через **NAT Gateway** (`yandex_vpc_gateway.nat_gateway`) для обновлений и интеграций, но не наоборот.  
  - Использует Route Table (`yandex_vpc_route_table.private`) для маршрутизации трафика через NAT Gateway.

- **NAT Gateway**  
  - Позволяет приватным ресурсам обращаться вовне (например, за обновлениями), не раскрывая их адреса.  
  - В Yandex Cloud используется Shared Egress Gateway, который обеспечивает исходящий интернет-трафик для приватных подсетей.

Такой дизайн соответствует требованиям **разделения зон безопасности** и снижает поверхность атаки.

## 3. Выбор размеров виртуальных машин и дисков

- **Compute Instance для портала (`yandex_compute_instance.portal`)**  
  - Платформа `standard-v1` с 2 ядрами и 4 ГБ RAM — достаточна для начальной нагрузки (портал + веб-приложение).  
  - Диск 20 ГБ для ОС и приложения.  
  - По мере роста числа пользователей можно масштабировать ресурсы (cores, memory) или перейти на autoscaling.

- **Managed PostgreSQL для доменного DWH (`yandex_mdb_postgresql_cluster.dwh`)**  
  - Класс `s2.medium` (2 vCPU, 8 GB RAM) и диск `100 ГБ` (network-ssd) выбраны как стартовая конфигурация для пилота.  
  - Поддерживает типичную нагрузку для первых доменных витрин (финтех, клиники).  
  - В дальнейшем можно масштабировать вертикально (увеличивая resource_preset_id и размер диска) или горизонтально (шардинг, отдельные хранилища по доменам).

Подход «начать с разумного минимума и масштабировать по фактической нагрузке» позволяет:

- Избежать переплаты за неиспользуемые ресурсы на раннем этапе.  
- Быстро адаптироваться к реальным требованиям по производительности.

## 4. Data Lake и доменное DWH

- **Object Storage бакет (`yandex_storage_bucket.data_lake`)**  
  - Служит **Data Lake** для сырых и подготовленных данных.  
  - Включено версионирование, что важно для:  
    - восстановления после ошибок;  
    - анализа изменений данных;  
    - соблюдения требований аудита.

- **Managed PostgreSQL как доменное хранилище**  
  - Подходит для первых доменных витрин (операционная и управленческая отчётность).  
  - Легко интегрируется с большинством BI-инструментов.  
  - Может быть дополнен колонночными DWH-сервисами в зависимости от выбранного облака.

Разделение на Data Lake и DWH:

- Позволяет хранить **сырые данные** в почти неограниченном объёме.  
- Даёт возможность строить различные витрины и модели поверх одних и тех же источников.

## 5. Безопасность: Security Groups

- **Security Group для портала (`yandex_vpc_security_group.portal_sg`)**  
  - Открывает только порт 443 (HTTPS) для внешнего мира.  
  - Все прочие порты закрыты, что снижает риск атак.

- **Security Group для DWH (`yandex_vpc_security_group.data_sg`)**  
  - Разрешает доступ к порту 5432 (PostgreSQL) только с `portal_sg`.  
  - Таким образом, БД не доступна напрямую из интернета, а только через портал и внутренние сервисы.

В реальной среде конфигурация должна быть дополнена:

- Централизованным IAM (Service Accounts и роли).  
- Шифрованием данных «на диске» и «в полёте» (KMS, TLS).  
- Журналированием и аудитом доступа к данным.

## 6. Декларативный подход и IaC (Infrastructure as Code)

Использование Terraform и декларативного описания инфраструктуры даёт следующие преимущества:

- **Воспроизводимость**  
  - Одинаковая конфигурация может быть развёрнута в нескольких окружениях (dev, test, prod).  
  - Снижается риск «дрейфа конфигурации» между средами.

- **Прозрачность и контроль изменений**  
  - Весь код хранится в Git, изменения проходят через code review и CI/CD.  
  - Команда видит полную историю эволюции инфраструктуры.

- **Автоматизация и скорость**  
  - Развёртывание новых окружений и ресурсов выполняется автоматически (`terraform apply`).  
  - Уменьшается доля ручных операций и связанных с ними ошибок.

- **Масштабируемость**  
  - При росте требований достаточно скорректировать параметры (типы инстансов, размеры дисков, количество подсетей и т. д.) и выполнить `plan/apply`.  
  - Конфигурация легко расширяется новыми ресурсами (стриминг, оркестраторы, дополнительные доменные хранилища).

## 7. Выбор Yandex Cloud

Выбор Yandex Cloud в качестве облачного провайдера обоснован следующими факторами:

- **Соответствие требованиям локализации данных**  
  - Данные остаются в российском дата-центре, что важно для медицинских и финансовых данных.

- **Управляемые сервисы**  
  - Managed PostgreSQL избавляет от необходимости администрировать БД вручную.  
  - Автоматические бэкапы и обновления повышают надёжность.

- **Экономическая эффективность**  
  - Конкурентные цены на вычислительные ресурсы и хранилища.  
  - Гибкая модель тарификации позволяет оптимизировать затраты.

- **Интеграция с экосистемой**  
  - Object Storage, Compute Cloud и Managed Databases хорошо интегрируются между собой.  
  - Поддержка Terraform провайдера обеспечивает удобство автоматизации.

## 8. Связь с бизнес-целями

- **Скорость вывода новых решений**:  
  - Быстрая подготовка инфраструктуры для доменов и портала самообслуживания.  
  - Уменьшение времени на согласование и реализацию изменений.

- **Качество медицинских и финансовых сервисов**:  
  - Надёжная и защищённая база для хранения доменных данных.  
  - Возможность быстро масштабировать ресурсы под реальные нагрузки.

- **Снижение затрат и рисков**:  
  - Оплата только за используемые ресурсы.  
  - Снижение человеческого фактора за счёт автоматизации и IaC.
