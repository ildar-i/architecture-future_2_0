# Задание 3 — Роадмап изменений для «Будущее 2.0»

## Горизонт: 12 месяцев

### Этап 0 (месяц 0–1) — Уточнение требований и подготовка платформы

- **Цели**:  
  - Зафиксировать ключевые бизнес-сценарии, KPI и домены.  
  - Выбрать целевой облачный провайдер и базовые технологии.
- **Ключевые шаги**:  
  - Уточнение требований по порталу самообслуживания и витринам (совместно с бизнесом).  
  - Определение доменов: Клиника, Финтех, ИИ, Партнёры, Управленческая отчётность.  
  - Выбор облачного DWH и Data Lake, согласование подхода к Data Mesh.  
  - Первичное моделирование целевой архитектуры (диаграммы контейнеров, DFD).  
- **Ответственные**: архитектор предприятия, владельцы доменов, ИТ-дирекция.  
  - **Результат**: согласованная целевая архитектура и список приоритетных сценариев.

### Этап 1 (месяц 1–3) — Запуск базовой платформы данных и инфраструктуры

- **Цели**:  
  - Поднять минимально жизнеспособную платформу данных в облаке.  
  - Заложить основу для IaC и CI/CD.
- **Ключевые шаги**:  
  - Развёртывание базовой инфраструктуры (VPC, сети, базы, хранилища) через Terraform.  
  - Настройка Data Lake и колонночного DWH.  
  - Внедрение IAM, шифрования, аудита доступа.  
  - Настройка CI/CD для инфраструктуры и базовых сервисов.  
- **Ответственные**: DevOps/Cloud-инженеры, архитекторы данных.  
  - **Результат**: платформа готова к приёму данных и созданию первых доменных витрин.

### Этап 2 (месяц 3–6) — Доменные хранилища и миграция критичных сценариев

- **Цели**:  
  - Вынести критичные сценарии из монолитного DWH в домены.  
  - Обеспечить первые бизнес-результаты для клиник и финтеха.
- **Ключевые шаги**:  
  - Создание доменных хранилищ для Клиники и Финтеха в облаке.  
  - Настройка потоков данных (CDC/стриминг и batch) из легаси-систем в платформу.  
  - Реализация витрин для ключевых отчётов по клиникам и финансам.  
  - Пилотное внедрение интеграционного слоя (API Gateway, стриминг).  
  - Ограниченная миграция исторических данных из DWH по приоритетным сценариям.  
- **Ответственные**: команды доменов Клиника и Финтех, архитекторы данных.  
  - **Результат**: первые отчёты и витрины работают на новой платформе, time-to-market сокращён.

### Этап 3 (месяц 6–9) — Портал самообслуживания и внедрение ИИ

- **Цели**:  
  - Запустить портал самообслуживания для доменов.  
  - Подключить ИИ-сервисы к платформе данных.
- **Ключевые шаги**:  
  - Разработка и запуск веб-портала самообслуживания (интеграция с BI).  
  - Конфигурация витрин для основных ролей (руководство, операционные менеджеры, продуктовые команды).  
  - Внедрение feature store и пайплайнов обучения моделей.  
  - Развёртывание первых моделей ИИ в продакшен с мониторингом качества.  
- **Ответственные**: команда портала, BI-команда, команда ИИ.  
  - **Результат**: бизнес-пользователи строят отчёты самостоятельно, первые модели ИИ дают измеримый эффект.

### Этап 4 (месяц 9–12) — Масштабирование и интеграция партнёров

- **Цели**:  
  - Подключить партнёрские домены (фарма, производители оборудования).  
  - Оптимизировать архитектуру и завершить ключевые миграции.
- **Ключевые шаги**:  
  - Интеграция с внешними партнёрами через API Gateway и стриминг.  
  - Расширение витрин отчётности для партнёрских сценариев (поставки, SLA, телеметрия).  
  - Оптимизация архитектуры с учётом фактической загрузки и новых требований.  
  - Подготовка плана по дальнейшему выводу из эксплуатации легаси DWH по остаточным сценариям.  
- **Ответственные**: команда партнёров, архитекторы, ИТ-дирекция.  
  - **Результат**: экосистема интегрирована, ключевые сценарии работают на новой архитектуре, риски легаси существенно снижены.

## Обоснование роадмапа

- **Выравнивание по целям бизнеса**: каждый этап даёт осязаемый результат (платформа, доменные витрины, портал, ИИ, партнёры), который поддерживает выручку, качество сервиса и скорость вывода новых продуктов.  
- **Управление рисками**: переход к новой архитектуре осуществляется поэтапно, без одномоментного отключения легаси.  
- **Фокус на доменах**: команды получают ответственность за свои домены и могут планировать развитие в рамках общей платформы.  
- **Использование IaC и облака**: обеспечивает воспроизводимость, масштабируемость и снижение затрат на сопровождение.
