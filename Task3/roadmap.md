# Задание 3 — Роадмап изменений для «Будущее 2.0»

## Горизонт: 12 месяцев

### Этап 0 (месяц 0–1) — Уточнение требований и подготовка платформы

- **Цели**:  
  - Зафиксировать ключевые бизнес-сценарии, KPI и домены.  
  - Выбрать целевой облачный провайдер и базовые технологии.
- **Ключевые шаги**:  
  - Уточнение требований по порталу самообслуживания и витринам (совместно с бизнесом).  
  - Определение доменов: Клиника, Финтех, ИИ, Партнёры, Управленческая отчётность.  
  - Выбор облачного DWH и Data Lake, согласование подхода к Data Mesh.  
  - Первичное моделирование целевой архитектуры (диаграммы контейнеров, DFD).  
- **Ответственные**: архитектор предприятия, владельцы доменов, ИТ-дирекция.  
- **Технологии из радара**:  
  - **Assess → Trial**: Data Mesh-подход (доменные витрины и владение данными в доменах).  
  - **Adopt**: выбор конкретного облачного провайдера и колонночного хранилища (BigQuery/Snowflake/Redshift).  
  - **Adopt**: выбор объектного хранилища (S3/GCS/Azure Blob) для Data Lake.
- **KPI и Definition of Done**:  
  - Согласована целевая архитектура (диаграммы контейнеров и DFD утверждены архитектурным комитетом).  
  - Определены 5 доменов с границами ответственности и владельцами.  
  - Выбран облачный провайдер и базовые технологии (DWH, Data Lake) с обоснованием.  
  - Сформирован список приоритетных бизнес-сценариев (минимум 10 сценариев с указанием доменов).  
  - Подготовлен план миграции с оценкой рисков и зависимостей.

### Этап 1 (месяц 1–3) — Запуск базовой платформы данных и инфраструктуры

- **Цели**:  
  - Поднять минимально жизнеспособную платформу данных в облаке.  
  - Заложить основу для IaC и CI/CD.
- **Ключевые шаги**:  
  - Развёртывание базовой инфраструктуры (VPC, сети, базы, хранилища) через Terraform.  
  - Настройка Data Lake и колонночного DWH.  
  - Внедрение IAM, шифрования, аудита доступа.  
  - Настройка CI/CD для инфраструктуры и базовых сервисов.  
- **Ответственные**: DevOps/Cloud-инженеры, архитекторы данных.  
- **Технологии из радара**:  
  - **Adopt**: Infrastructure as Code (Terraform) для управления облачной инфраструктурой.  
  - **Adopt**: Облачное колонночное хранилище (BigQuery/Snowflake/Redshift) для доменных хранилищ.  
  - **Adopt**: Облачное объектное хранилище (S3/GCS/Azure Blob) как Data Lake (raw/processed/curated слои).  
  - **Adopt**: CI/CD для приложений и пайплайнов данных (GitLab CI/GitHub Actions/Jenkins).  
  - **Adopt**: Мониторинг и логирование (Prometheus/Grafana, ELK/EFK, облачные решения).
- **KPI и Definition of Done**:  
  - Инфраструктура развёрнута через Terraform (100% инфраструктуры описано в коде).  
  - Data Lake настроен с зонами raw/processed/curated, доступен для загрузки данных.  
  - Колонночное хранилище развёрнуто и доступно для создания витрин.  
  - IAM настроен с ролями для доменов, шифрование данных включено.  
  - CI/CD pipeline работает для инфраструктуры (автоматический деплой через merge в main).  
  - Мониторинг настроен: доступность платформы > 99.5%, метрики инфраструктуры собираются.  
  - Документация по инфраструктуре и процессам обновлена.

### Этап 2 (месяц 3–6) — Доменные хранилища и миграция критичных сценариев

- **Цели**:  
  - Вынести критичные сценарии из монолитного DWH в домены.  
  - Обеспечить первые бизнес-результаты для клиник и финтеха.
- **Ключевые шаги**:  
  - Создание доменных хранилищ для Клиники и Финтеха в облаке.  
  - Настройка потоков данных (CDC/стриминг и batch) из легаси-систем в платформу.  
  - Реализация витрин для ключевых отчётов по клиникам и финансам.  
  - Пилотное внедрение интеграционного слоя (API Gateway, стриминг).  
  - Ограниченная миграция исторических данных из DWH по приоритетным сценариям.  
- **Ответственные**: команды доменов Клиника и Финтех, архитекторы данных.  
- **Технологии из радара**:  
  - **Trial → Adopt**: Data Mesh-подход (доменные витрины и владение данными в доменах) для Клиники и Финтеха.  
  - **Adopt**: Стриминговая платформа (Kafka/Pub/Sub/Event Hubs) для событий клиник и финтех-транзакций.  
  - **Adopt**: API Gateway + REST/gRPC для интеграции доменов.  
  - **Trial**: Event-driven architecture для ключевых доменов (реакция на события в реальном времени).  
  - **Adopt**: Облачное колонночное хранилище для доменных витрин.
- **KPI и Definition of Done**:  
  - Доменные хранилища созданы для Клиники и Финтеха (минимум 2 хранилища, доступны для загрузки данных).  
  - Потоки данных настроены: CDC/стриминг работает для критичных источников (задержка < 5 минут).  
  - Реализовано минимум 5 ключевых витрин (2 для клиник, 2 для финтеха, 1 общая).  
  - API Gateway развёрнут, минимум 3 API endpoint доступны для доменов.  
  - Миграция исторических данных: минимум 1 год данных загружен в платформу.  
  - Первые отчёты работают на новой платформе: время генерации отчёта сокращено с часов до < 10 минут.  
  - Бизнес-пользователи подтвердили корректность данных в новых витринах (accuracy > 99%).

### Этап 3 (месяц 6–9) — Портал самообслуживания и внедрение ИИ

- **Цели**:  
  - Запустить портал самообслуживания для доменов.  
  - Подключить ИИ-сервисы к платформе данных.
- **Ключевые шаги**:  
  - Разработка и запуск веб-портала самообслуживания (интеграция с BI).  
  - Конфигурация витрин для основных ролей (руководство, операционные менеджеры, продуктовые команды).  
  - Внедрение feature store и пайплайнов обучения моделей.  
  - Развёртывание первых моделей ИИ в продакшен с мониторингом качества.  
- **Ответственные**: команда портала, BI-команда, команда ИИ.  
- **Технологии из радара**:  
  - **Adopt**: Современный BI-инструмент (Power BI/Tableau/Looker) для портала самообслуживания.  
  - **Trial**: Feature Store для ИИ (Feast/Vertex AI Feature Store/SageMaker Feature Store).  
  - **Adopt**: ML/AI-стек на Python (scikit-learn, TensorFlow, PyTorch) + управляемые сервисы в облаке.  
  - **Trial**: MLOps-платформы (Kubeflow, MLflow, Vertex AI, SageMaker) для управления жизненным циклом моделей.  
  - **Adopt**: Мониторинг и логирование для ИИ-моделей (метрики качества, дрифт, производительность).
- **KPI и Definition of Done**:  
  - Портал самообслуживания запущен и доступен для бизнес-пользователей (uptime > 99%).  
  - Настроено минимум 3 роли с разными уровнями доступа (руководство, менеджеры, продуктовые команды).  
  - Минимум 10 витрин доступны через портал для самостоятельного построения отчётов.  
  - Feature Store развёрнут, минимум 20 фич доступны для моделей.  
  - Развёрнуто минимум 2 модели ИИ в продакшен (1 для медицины, 1 для финтеха).  
  - Мониторинг моделей настроен: метрики качества (accuracy, precision, recall) отслеживаются в реальном времени.  
  - Бизнес-эффект от ИИ измерим: улучшение KPI на минимум 5% (например, точность диагностики, конверсия кредитов).  
  - Минимум 50% бизнес-пользователей используют портал для самостоятельного построения отчётов (без обращения к BI-команде).

### Этап 4 (месяц 9–12) — Масштабирование и интеграция партнёров

- **Цели**:  
  - Подключить партнёрские домены (фарма, производители оборудования).  
  - Оптимизировать архитектуру и завершить ключевые миграции.
- **Ключевые шаги**:  
  - Интеграция с внешними партнёрами через API Gateway и стриминг.  
  - Расширение витрин отчётности для партнёрских сценариев (поставки, SLA, телеметрия).  
  - Оптимизация архитектуры с учётом фактической загрузки и новых требований.  
  - Подготовка плана по дальнейшему выводу из эксплуатации легаси DWH по остаточным сценариям.  
- **Ответственные**: команда партнёров, архитекторы, ИТ-дирекция.  
- **Технологии из радара**:  
  - **Trial → Adopt**: Data Mesh-подход расширен на домен Партнёры.  
  - **Adopt**: API Gateway + REST/gRPC для B2B-интеграций с партнёрами.  
  - **Adopt**: Стриминговая платформа для телеметрии оборудования и событий поставок.  
  - **Trial**: Service Mesh (Istio/Linkerd) для критичных доменов (финтех, интеграция с партнёрами).  
  - **Assess**: GraphQL для агрегированных запросов витрины (опционально, для сложных выборок).
- **KPI и Definition of Done**:  
  - Домен Партнёры создан и интегрирован с платформой данных.  
  - Минимум 2 партнёра подключены через API Gateway (фарма и производитель оборудования).  
  - Потоки данных от партнёров настроены: телеметрия и события поставок поступают в реальном времени.  
  - Расширено минимум 3 витрины для партнёрских сценариев (поставки, SLA, телеметрия).  
  - Оптимизация архитектуры завершена: время генерации отчётов < 5 минут для 90% запросов.  
  - Минимум 80% критичных сценариев мигрированы с легаси DWH на новую платформу.  
  - План вывода из эксплуатации легаси DWH подготовлен и согласован с бизнесом.  
  - Экосистема интегрирована: все домены (Клиника, Финтех, ИИ, Партнёры, Управленческая отчётность) работают на новой архитектуре.

## Обоснование роадмапа

- **Выравнивание по целям бизнеса**: каждый этап даёт осязаемый результат (платформа, доменные витрины, портал, ИИ, партнёры), который поддерживает выручку, качество сервиса и скорость вывода новых продуктов.  
- **Управление рисками**: переход к новой архитектуре осуществляется поэтапно, без одномоментного отключения легаси.  
- **Фокус на доменах**: команды получают ответственность за свои домены и могут планировать развитие в рамках общей платформы.  
- **Использование IaC и облака**: обеспечивает воспроизводимость, масштабируемость и снижение затрат на сопровождение.
